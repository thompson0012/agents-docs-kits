**v1.6.3 → v2.0 心智模型對照表**

---

## 1. project-initializer（專案初始化）

**v1.6.3 心智模型：** 一個獨立的 worker phase，負責建立或修復 `docs/live/*` 控制平面檔案，種子 reference docs，處理 records 追溯。

**v2.0 對應：**
→ **不復存在。** 拆分為兩個簡單機制：
- `docs/scripts/init.sh` — 建立目錄 + 寫入 `plan.md` + `tracked-work.json` 預設值（bash 腳本，非 LLM agent）
- **Orchestrator cold start** — 讀取 AGENTS.md → plan.md → tracked-work.json，若無活躍 workstream，提示人類建立第一個

**歸類原因：** 初始化不需要一個 LLM worker phase。它是檔案系統操作，腳本就能完成。原設計把一個 bash script 的事放大成一個完整 worker prompt（187 行）。

---

## 2. generator-brainstorm（發散思維階段）

**v1.6.3 心智模型：** 一個獨立的 pre-sprint、非 runnable phase。將模糊想法精煉成 `ideas.md` 中的結構化探索。可將合格候選晉升為 `pending` backlog item。包含自我攻擊步驟（檢查 complexity signals、scale appropriateness）。可選擇性建立 scoped `docs/records/*` 頁面。

**v2.0 對應：**
→ **吸收到 thesis phase 的前半部。** thesis worker 的「ground in prior evidence」+「form the narrowest honest claim」+「self-attack（can this be falsified?）」三步驟直接對應 v1.6 brainstorm 的核心行為。

**歸類原因：** 原 brainstorm 和 proposal 是兩個 phase，但它們做的是同一件事的不同階段——先澄清問題，再形成主張。v2.0 合併為一個命題形成步驟，減少 pipeline 的 hop 數量。complexity signals 和 scale guide 的 self-attack 檢查保留在 thesis worker 的「self-attack」步驟中，但不以獨立 phase 出現。

---

## 3. generator-proposal（提案階段）

**v1.6.3 心智模型：** 一個獨立的 worker phase。將一個 `pending` backlog item 轉化為 bounded sprint proposal（`sprint_proposal.md`）。包含：問題描述、範圍/禁止檔案、AC 驗收標準（含 stateful/reversible 標記）、風險/假設、結構化 JSON task decomposition（含 symbol 宣告）、roadmap 切片分解。

**v2.0 對應：**
→ **拆散到兩個 phase：thesis + contract。**
- thesis 吸收了「形成主張」的部分（命題、前提、邊界、預期結果）
- contract 吸收了「定義建造邊界」的部分（允許/禁止檔案、AC 驗收標準、風險、非目標）
- JSON task decomposition（symbol 宣告）→ **捨棄。** 這是一個 machine-verifiable formalization，但 oracle 審查指出它被 LLM 產出後的真實價值有限（LLM 自行決定如何 partition，結構檢查通過不代表架構品質）

**歸類原因：** 原 proposal 做了兩件事：說「我認為應該這樣」（命題）和說「我會改這些檔案」（合約）。v2.0 把它們分到兩個不同的 layer——命題屬於 Direction Layer，合約屬於 Action Layer。這讓純推理工作流（不需要合約的）可以停在 thesis，而不必走完 proposal 管道。

---

## 4. evaluator-contract-review（合約審查階段）

**v1.6.3 心智模型：** 一個獨立的 adversarial worker phase。攻擊 proposal，產出 `contract.md`（批准）或 `review_feedback.md`（拒絕並要求修訂）。最多 `max_proposal_revisions` 次（預設 3），耗盡後 escalate_to_human。**這是 v1.6 的第一道 adversarial gate。**

**v2.0 對應：**
→ **吸收到 challenge phase。** challenge worker 做同樣的事——adversarially test 一個主張（thesis），但委派給 oracle 或 council 而非自己做判斷。原 evaluator 是「審查 proposal → 產出合約」，challenge 是「審查 thesis → 產出判決」。

差異：
- 原 evaluator 的產物是 `contract.md`（一個執行邊界）。challenge 的產物是 `challenge.md`（一個結構化判決 + comprehension gate）
- 原 evaluator 直接做判斷。challenge 委派給 oracle/council，自己不判斷
- 原 evaluator 有 revision loop（`proposal_revision_required` → 修訂 → 再審查）。v2.0 的等價機制是 spiral turn（challenge 揭示的 gap 經 response → synthesis 處理後，若框架不自洽，回到 thesis）

**歸類原因：** 原設計有兩道 adversarial gate（evaluator 審合約 + reviewer 審執行），v2.0 合併為一道（challenge 審命題 + audit 審執行），減少了形式主義。但 v2.0 補了一個關鍵改善：challenge 明確委派給 oracle/council，承認 LLM 不善於自我對抗的事實。

---

## 5. generator-execution（執行階段）

**v1.6.3 心智模型：** 一個獨立的 worker phase。實現已批准的合約。流程：restate contract → validate contract structure（透過 `validate_contract.py`）→ derive system view from task decomposition → implement within bounds → capture verification evidence in `runtime.md` → build/startup triage → write `handoff.md`。支援 retry（`attempt_count`、`clean_restore_ref`）。

**v2.0 對應：**
→ **拆散到兩個 phase：contract + build。**
- contract worker 吸收「定義邊界」的職責（v1.6 的 generator-execution 第一步 "restate contract" 和 entry checks）
- build worker 吸收「實作」的職責（implement within bounds、build/startup triage、write handoff.md）

被移除的部分：
- `runtime.md` → **捨棄。** v2.0 的 build worker 直接寫 `handoff.md`，裡面包含 reproduction steps + acceptance trace，不再需要一個獨立的 runtime 中間 artifact
- `validate_contract.py` 腳本驗證 → **捨棄。** contract worker 透過直接檢查實際程式碼來內嵌驗證，而非透過外部腳本
- System view from task decomposition → **捨棄。** task decomposition JSON 已不再存在

**歸類原因：** 原 execution phase 太長（337 行 SKILL.md），做了太多事。v2.0 把「定義範圍」和「寫程式」分離為兩個 phase，反映它們實際上是不同的心智操作。

---

## 6. adversarial-live-review（對抗審查階段）

**v1.6.3 心智模型：** 一個獨立的 adversarial worker phase。**這是 v1.6 的第二道 adversarial gate。** 從 `handoff.md` 重現結果，比對合約，產出 PASS/FAIL/BLOCKED 判決 + P0-P3 findings。必須驗證 before/action/after 狀態轉換，拒絕靜態截圖或 hardcoded 狀態。449 行 SKILL.md。

**v2.0 對應：**
→ **被 audit phase 取代，但本質相同。** PASS/FAIL/BLOCKED 判決保留。P0-P3 findings 保留。狀態轉換驗證保留。靜態證據拒絕保留。

關鍵差異：
- audit 新增 **deeper insight assessment**——不只是判斷「程式碼對不對」，而是判斷「這個結果是否揭示底層 thesis 有問題」。這是 spiral turn 的觸發器
- audit 的 SKILL.md 是 107 行（vs. v1.6 的 449 行），減少了 76%
- 移除了：coverage metadata（`coverage_status`、`criteria_total`、`criteria_checked`、`convergence_status`）、duplicate_of 關聯、acceptance id accounting——這些是 v1.6 對抗審查的 formalization，在實踐中增加了審查器的 overhead 但沒有改善判決品質

**歸類原因：** v2.0 保留了對抗分離的核心不變量（Generator ≠ Auditor），但承認 LLM 審查的誠實性限制，去掉了假裝它能做到完美形式驗證的 ceremony。

---

## 7. state-update（狀態更新階段）

**v1.6.3 心智模型：** 一個獨立的 worker phase。419 行 SKILL.md。在 review 後協調全局面狀態。職責包括：更新 `tracked-work.json`、刷新 `current-focus.md`、更新 `roadmap.md`、附加 `progress.md`、歸檔 PASS sprint 到 `docs/archive/`、切換 `evidence_path`、清除 runnable active sprint、隊列 `compound_pending_feature_ids`、處理 retry metadata（`attempt_count`、`clean_restore_ref`）、處理 parked human gates。

**v2.0 對應：**
→ **吸收到 orchestrator 的 post-audit routing 邏輯中。** 不再是獨立的 worker phase。當 audit.md 存在時，orchestrator 直接：
- PASS + no deeper insight → 歸檔（移動檔案到 archive、更新 tracked-work.json、更新 plan.md）
- PASS + deeper insight → spiral turn（depth++、reset to thesis）
- FAIL/BLOCKED → retry 或 human gate

**歸類原因：** state-update 做的是機械性的狀態同步工作——讀一個判決，更新 JSON 欄位，移動檔案。這不需要一個 LLM agent。它是 pipeline 中的一個形式主義 hop。v2.0 把它簡化為 orchestrator 內的幾行 routing 邏輯。

---

## 8. compound-capture（複利學習階段）

**v1.6.3 心智模型：** 一個獨立的 non-runnable phase。消費 `compound_pending_feature_ids`，從 decisive sprint outcomes 提取跨 sprint 學習，寫入 `memory.md`。可建立/更新 `docs/records/*` 頁面。可更新 reference docs。**在每次狀態更新後強制 drain（先 compound，再開新 sprint）。**

**v2.0 對應：**
→ **吸收到歸檔動作中。** 不再是獨立 phase。orchestrator 歸檔 PASS sprint 時，將 lessons 直接 append 到 `plan.md` 的 `## Lessons` 區段。無隊列、無獨立 phase、無強制 serialization gate。

**歸類原因：** oracle 審查指出 compound-capture 的成本（serialization bottleneck，每次必須 drain 才能開新工作）超過其價值。提取跨 sprint 學習是真實需求，但一個獨立 phase 是 over-engineered。在歸檔時附加一段文字到 plan.md 就足夠了。

---

## 9. 被移除的子系統

| v1.6.3 子系統 | v2.0 對應 |
|---|---|
| `docs/records/*`（含生命週期、metadata、domain 分類） | → **捨棄。** 需持久化的 insight 直接寫入對應 artifact（thesis/challenge/response/synthesis/audit） |
| `docs/scripts/*.py`（6 個維護腳本） | → **捨棄。** 路由邏輯內嵌於 SKILL.md 和 orchestrator |
| `using-agents-stack/scripts/`（8 個驗證/分配腳本） | → **捨棄。** 3,138 行 Python 被 orchestrator 內嵌邏輯取代 |
| 12 層證據優先級 | → 簡化為 4 層（audit > handoff > contract > direction/method artifacts） |
| 五要素 dispatch packet 形式主義 | → 簡化為「SKILL.md 路徑 + workstream ID + artifact 路徑」 |
| 五類失敗分類學 | → 路由邏輯內嵌（無平行分類系統） |
| 三層驗證鏈（A→B→C） | → 保留 A→B（generator→auditor），第三層不加 |
| P0-P2 硬化清單 | → invariant 自身就是硬化 |
| `progress.md`（append-only ledger） | → plan.md 的 What 區段反映當前狀態 |
| `roadmap.md` + `current-focus.md` + `ideas.md` 三分離 | → 合併為 `plan.md` 的三個區段 |

## 總體歸納：v1.6 的 8 個 worker phase → v2.0 的 3 層 7 phase

```
v1.6.3 (線性管道)                 v2.0 (螺旋三層)

project-initializer  ─────────→   init.sh + orchestrator cold start
generator-brainstorm ─┐
                      ├──────→   thesis (Direction Layer)
generator-proposal  ──┘
evaluator-contract    ───────→   challenge (Direction Layer)
  -review
(gap: no synthesis)   ───────→   response → synthesis (Method Layer)  ★ 新增
generator-execution   ─┬─────→   contract (Action Layer)
                       └─────→   build (Action Layer)
adversarial-live      ───────→   audit (Action Layer)
  -review
state-update          ───────→   orchestrator post-audit routing
compound-capture      ───────→   plan.md Lessons (歸檔時附加)

(gap: no spiral)      ───────→   spiral turn (depth++, back to thesis) ★ 新增
```

**v2.0 新增的兩個能力（v1.6 沒有的）：**
1. **Method Layer（response → synthesis）** — v1.6 從 challenge 直接跳到 execution，缺少「如何回應挑戰發現的缺口」的設計階段
2. **Spiral turn** — v1.6 的 retry 只允許在同一合約內重試。v2.0 允許 audit 發現的 deeper insight 觸發返回 thesis 的新螺旋（depth++）

**v2.0 從 16 個 phase 減至 7 個 phase 的原理：**
- 合併了做同一件事的不同步驟（brainstorm + proposal → thesis）
- 拆分了做太多事的 phase（execution → contract + build）
- 將機械性工作從 LLM worker 移到 orchestrator 內嵌邏輯（state-update、compound-capture）
- 將純檔案操作從 phase 移到 bash 腳本（project-initializer → init.sh）
- 承認 LLM 的限制，去掉了形式主義驗證層（task decomposition JSON validation、coverage metadata accounting）
